


# リポジトリ基本ルール

このリポジトリは、複数のミニアプリケーションを統合・管理するための構成になっています。

## 1. ディレクトリ構造とアプリの定義

- 各アプリケーションはルート直下の個別のディレクトリに配置します。
- **アプリとして認識される条件**: ディレクトリ内に `index.html` が存在し、ルートの `apps.json` にエントリが存在すること。
- **除外対象**: `.` または `_` で始まるディレクトリ（例: `_template`, `.git`）および `node_modules` はアプリとして認識されません。
- **依存関係管理**: 各アプリディレクトリ内に個別の `package.json` は作成・配置しません。依存関係はすべてルートの `package.json` で管理してください。

## 2. 新規アプリの作成

- 新規アプリを作成する際は、ルートにある `_template` ディレクトリをコピーして使用してください。
- `create-app` スキルを使用して作成することを推奨します。
- ディレクトリ名およびアプリの識別子には、英数字とハイフンのみ（`kebab-case`）を使用してください。

## 3. ビルドとプロジェクト設定

- `vite.config.ts` はディレクトリ構成を動的にスキャンしてビルド設定を生成するため、**新しいアプリを追加する際に手動で編集する必要はありません**。
- `apps.json`（ルート）には以下の情報を含めてください：
  - `"title"`: 人間が読める形式のアプリ名（例：`My New App`）。
  - `"description"`: 「どんなゲームなのか、ゲームのコアとなる面白さ」が伝わる簡潔な説明。
  - `"image"`: ポータル用メイン画像（例：`"/thumbnails/<app-name>.png"`）。
- `index.html` の更新：
  - アプリディレクトリ内の `index.html` の `<title>` タグを、`apps.json` の `"title"` と一致させてください。

## 4. ポータル（index.html）

- ルートの `index.html` は各アプリへのポータル画面です。
- `<!-- PORTAL_CARDS -->` というコメント位置に、各アプリの `app.json` から抽出されたタイトルと説明文、画像が自動挿入されます。このコメントを削除しないでください。

## 5. 静的アセットの管理（ハイブリッド方式）

- **ポータル用サムネイル画像**: ルートの `public/thumbnails/` ディレクトリに配置し、`apps.json` の `"image"` には `"/thumbnails/ファイル名.png"` のようにルート相対パスを指定してください。
- **共通アセット**: `src/assets/shared/` に配置し、Vite エイリアス `@shared/` を使用して参照してください（例: `@shared/logos/logo_x.svg`）。
- **ゲーム個別の要素**: 各アプリディレクトリ内の `src/assets/` に配置してください。TypeScript内で `import` して使用するか、ゲームの `index.html` から相対パス（`./src/assets/...`）で参照することで、Viteが適切にビルド・ハッシュ化を行います。ただし、共通ロゴなどの共有資産については上述の `@shared` エイリアスを使用することを推奨します。

## 6. Git 操作の制限

- **自動コミットの禁止**: ユーザーから明示的な指示（「コミットして」「保存して」など）がない限り、`git commit` や `git push` を自動的に実行しないでください。
- コミットを行う際は、`commit` スキルを使用することを推奨します。

## 7. コード品質、CI、および検証

- **検証の必須化**: 全てのコード変更後、およびプロジェクトの終了前には、必ずルートディレクトリで `make ci` を実行し、全チェックが成功することを確認してください。
  - `make ci` は、型チェック (`check-ts`)、Lint (`ts-check-diff`, `html-check-diff`)、カスタムルール (`check-ts-rules`) を実行します。
- **ビルドの確認**: ルートディレクトリで `npm run build` を実行し、ビルドエラーが発生しないことを確認してください。
- **LLM用コンテキスト生成**: プロジェクトを俯瞰した分析が必要な場合は、`make repomix` を使用して統合ファイルを生成してください。
- **品質基準**:
  - `any` 型や `@ts-ignore` の使用は原則禁止です（`check-ts-rules.py` で検証されます）。
  - エラーやワーニングを放置したままタスクを完了としないでください。

# Architecture Map 運用ルール

このリポジトリでは `architecture-map.md`（またはそれに類するドキュメント）を、コードベース理解の「一次情報（Source of Truth）」として扱います。

## 1. 必ず最初に読む

新しい作業を始める前に、リポジトリの構造を把握するために既存のドキュメントを読んでください。

## 2. 構造変更があったら更新する

次のいずれかに該当する変更を行った場合、関連するアーキテクチャドキュメント（または `README.md`）を更新してください。

- **新しいディレクトリや主要モジュールの追加**
- **エントリポイントやビルドプロセスの変更**
- **主要なデータフローや依存関係の変更**
- **CI / 検証ルールの追加・変更**

## 3. 更新時の注意

- 既存の構成を維持し、必要な箇所だけを追記・修正してください。
- 「どのファイルが何を担当しているか」を明確に記述してください。




- GitHub 上の操作には `gh` コマンドを利用できます。

# 言語ルール（最優先）
- すべての回答、思考プロセス、アーティファクト（プラン、タスク、ウォークスルー等）、およびコード内のコメントは、例外なく**厳密な日本語**で記述してください。
- 英語での出力はシステム上の制約として禁止されています。
- ユーザーから個別の指示がない限り、日本語以外での出力は行わないでください。
